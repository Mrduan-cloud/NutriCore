# NutriCore 硬件选型 / 成本 / ROI 对照

> 这份文档回答三个问题：
> 1. 不同公司阶段该用什么配置？
> 2. 各档硬件的真实采购成本是多少？
> 3. 我的业务能否覆盖硬件投入？
>
> 默认推荐：**2× RTX 4090 24G + Qwen2.5-32B-Instruct-AWQ + vLLM Tensor Parallel(TP=2)**。详细启动方式见 [`DEPLOYMENT.md` §2.5 / §3.4](DEPLOYMENT.md)。

## 3. ROI 测算 — NutriCore 能不能覆盖成本？

按你简历背景（杭州鑫东田，对接消费品制造 + 医疗健康 B 端客户），典型营收：

| 营收模式 | 单笔金额 | 覆盖 2×4090（5w）需要 | 覆盖 2×A100（30w）需要 |
|---|---|---|---|
| B 端私有化交付一次性 | 30–80 万 / 项目 | **1 个客户就回本** | 1 个客户也能覆盖 |
| B 端 SaaS 订阅 | 5k–3w / 月 / 客户 | **3–5 个客户/年** | 15–20 个客户/年 |
| C 端订阅（AI 营养师 / 个性化方案）| 30–100 元/月/用户 | 500–1,500 付费用户 | 3,000–8,000 付费用户 |

**结论**：2× RTX 4090 这套配置对小公司是「**接到第一个客户就回本，剩下都是利润**」的甜蜜区。如果业务规模做到稳定 5–10 家 B 端客户再考虑升级到 A100。

## 4. 为什么 NutriCore 默认选 32B-AWQ + 2× 4090 而不是 7B + 单卡

| 选项 | 智商上限 | 显存需求 | 整机成本 | 适合场景 |
|---|---|---|---|---|
| 7B FP16 / Q4 | 较低 | 4–16G | 单卡 4090 ~2w | 简单对话 / 工具调用 |
| 14B AWQ | 中等 | ~10G | 单卡 4090 ~2w | 兼顾速度与质量 |
| **32B AWQ + TP=2** | **较高** | **~20G/卡** | **2× 4090 ~5w** | **复杂多 Agent 编排、引用核验、医疗营养场景需要稳定推理质量** |
| 70B AWQ | 高 | ~40G | 4× 4090 / 双 A100 | 投入成本 4–6 倍，边际收益有限 |

**32B AWQ + TP=2 是消费级硬件能够稳定支撑「中等智商 + 多 Agent 编排」的最优甜点**：
- 7B 智商不够，多 Agent 链路上的意图识别 / 引用核验经常翻车
- 70B 需要 A100，超出小公司预算
- 32B AWQ 显存 ~20G，TP=2 把负载分到 2 张卡，单卡 24G 显存还有 ~3G KV cache 余量

## 5. 如果客户预算高 / 业务规模大，升级路径

```
[当前] 2× RTX 4090 + Qwen2.5-32B-AWQ
   ↓ 客户量增长 / 并发需求
[升级 1] 2× L20 48G 或 L40S 48G → 跑 32B FP16，吞吐 +2x
   ↓ 跨节点 / 需要 NVLink
[升级 2] 2× A100 80G → 跑 70B AWQ，质量上一个台阶
   ↓ 出海客户 / 头部企业级
[升级 3] 4-8× H100/H800 集群 → 跑 72B FP16 或 405B MoE
```

业务代码层不动，只需要换 `LLM_BASE_URL` 和 `LLM_MODEL`，调度由 vLLM 内部完成。这是 OpenAI 兼容协议的好处。

## 6. 面试时怎么讲这套配置（话术）

> 「我们公司是 25 人左右的小团队，硬件预算有限，所以做了模型选型权衡：用 Qwen2.5-32B-Instruct 而不是 7B 是因为多 Agent 编排里意图识别和引用核验对推理质量比较敏感；用 AWQ Int4 量化把显存从 ~64G 压到 ~20G，配合 vLLM 的 Tensor Parallel TP=2 切到 2 张 4090 上跑。整机成本控制在 5 万左右，已经能支撑单一客户 SaaS 私有化交付。客户预算高的时候可以无感升级到 L40S 或 A100，因为我们走的是 OpenAI 兼容协议，业务代码不用改。」

这套话术传达三个信号：
1. 你在小公司真实做过落地
2. 你懂模型选型 / 量化 / 张量并行
3. 你考虑过成本与业务的匹配
